# 实用工具决策台设计

## 目标与范围

将 `tools.html` 从三个互不关联的计算器，重构为围绕“买、攒、换”的 CS 饰品购买决策台。页面只使用用户输入的本地草稿数据，不接入第三方价格或账户资产，也不改变目录、搭配、库存和开箱页面。

目标是让用户在每个工具完成计算后都能立即知道下一步：可否按当前价格购买、愿望清单该先买什么、以及距离换购目标还差多少。

## 体验结构

### 1. 决策总览

首屏保留工具页标题，但将右侧两个静态数字卡扩展为“我的购买面板”：可用储蓄、愿望清单缺口和换购资金三个状态。面板根据现有草稿自动给出一条主行动建议：

- 愿望清单为空时，引导添加第一个目标；
- 愿望资金不足时，引导优先补足最近可完成的目标；
- 已有目标且资金充足时，引导先评估该物品的到手价；
- 有换购目标且资金不足时，提示差额和需要补充的资金。

### 2. 买入评估

保留标价、手续费、优惠和预算四项输入，结果从四个纯数字改为“决策结论 + 明细”。结论显示“预算内，可考虑下单”或“超出预算，建议等待/议价”，并醒目展示到手价、可接受最高标价、超支或节省金额与手续费。输入使用默认示例值，并提供一键填充示例，以避免初始空状态。

### 3. 愿望清单计划

保留每行“名称 + 价格”的简洁输入格式，增加购买排序策略：默认按价格从低到高，优先显示最接近当前储蓄、最可完成的一件。结果区显示总价、当前可买数量、总缺口、按月储蓄预计月数和“下一件建议”。列表每项显示相对于已有储蓄的完成状态，而不再只是名称和价格表格。

### 4. 出售换购

保留“卖出物品 + 手续费 + 补款 + 目标价”的输入方式。结果区明确区分卖出总额、扣费、可用资金、目标差额，并用进度条表现覆盖度。结论提示“已可换购”或“仍需补足 ¥X”，防止负数差额造成歧义。

## 状态与交互

- 每个工具保持独立表单，提交只刷新工具根节点，不导航、不请求网络。
- 输入草稿继续保存在现有 `appState` 中；新增的“是否使用示例”和“决策文案”均由已有草稿即时派生，不新增持久化或服务端状态。
- 愿望清单与换购所得共享到顶部总览，购买评估独立，避免三种资金含义被错误合并。
- 为每个面板提供语义化标题、状态文案和 `aria-live` 结果区域；移动端将结果纵向堆叠，按钮保持易触达尺寸。

## 实现边界

修改范围为 `app.js`、`styles.css`、`tools.html`（仅在必须更新资源版本号时）以及 `scripts/check-practical-tools.mjs`。维持已有导航、语言运行时和表单 ID；不删除现有计算函数的兼容入口，而是在其结果之上添加决策派生数据和新的呈现结构。

## 验证

- 扩展实用工具静态检查，覆盖总览、示例填充、购买结论、愿望优先项和换购结论的关键标识。
- 运行 `npm run test:practical-tools`。
- 在桌面和移动视口检查：首屏总览、三块工具的空状态与示例状态、结果层级、表单可用性和小屏单列布局。
- 检查 `git diff`，确认没有覆盖用户现有的无关改动。

## 自检

范围限于工具页；所有推荐均基于用户输入的本地计算，未暗示实时价格或投资建议。资金口径保持分离：储蓄仅用于愿望清单，出售净额仅用于换购，避免双重计算。
